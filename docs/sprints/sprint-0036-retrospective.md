# Sprint 36 Retrospective

**Delivered**: v0.3.13 — Ash.StateMachine extractor.
**Date**: 2026-04-01

## What was delivered

1. **Dev dependencies**: `ash` (~~> 3.0) and `ash_state_machine` (~~> 0.2) added as dev/test deps. Pulls in ~12 transitive deps (ecto, reactor, splode, etc.).

2. **`TLX.Extractor.AshStateMachine`** — simplest extractor yet. Uses `AshStateMachine.Info` runtime introspection to read states, transitions, initial states directly. No AST walking needed. Handles `:*` wildcard expansion, deprecated state exclusion, and graceful errors for non-Ash modules.

3. **`mix tlx.gen.from_ash_state_machine`** — mix task with pattern/codegen output.

## What went well

- The `AshStateMachine.Info` API (generated by Spark.InfoGenerator) provides exactly the structured data we need. The extractor is ~110 lines — half the size of other extractors.
- All transitions are `:high` confidence since the DSL is fully declarative.
- Wildcard expansion correctly excludes deprecated states.

## What could improve

- Ash resources require a full Domain + actions setup even for test fixtures. The test module definitions are verbose (~160 lines for 3 resources).
- `AshStateMachine.Info.state_machine_initial_states/1` raises `ArgumentError` on non-Spark modules instead of returning an error tuple — needed a rescue clause.
- The Inspect protocol consolidation warnings in tests are cosmetic but noisy.

## Numbers

- Tests: 340 unit + 87 integration
- New code: 1 extractor (~110 lines), 1 mix task (~100 lines), 1 test file (7 tests)
- New deps: ash, ash_state_machine + 12 transitive (dev/test only)
